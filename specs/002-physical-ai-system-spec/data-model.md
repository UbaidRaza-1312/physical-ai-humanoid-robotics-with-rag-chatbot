# Data Model: Physical AI System

This document defines the key data entities used within the Physical AI system. These are not database models but rather in-memory data structures, primarily exchanged as ROS 2 messages.

## 1. TaskGraph

Represents a plan generated by the cognitive planner to fulfill a user's command. It is a sequence of actions with parameters.

- **Type**: ROS 2 Message (`humanoid_planning/TaskGraph.msg`)
- **Fields**:
    - `uint32 graph_id`: A unique identifier for this task graph.
    - `humanoid_planning/Action[] actions`: An array of actions to be executed sequentially.

---

## 2. Action

Represents a single step within a `TaskGraph`.

- **Type**: ROS 2 Message (`humanoid_planning/Action.msg`)
- **Fields**:
    - `uint32 action_id`: A unique ID for this step within the graph.
    - `string type`: The type of action to perform (e.g., "navigate", "detect", "grasp", "speak").
    - `string[] params`: A list of parameters for the action (e.g., for "detect", params might be `["bottle", "red"]`).

---

## 3. HumanoidSystem (Conceptual)

This is a conceptual entity representing the entire robot. It is not a single data structure but is composed of many smaller status and control messages published across the ROS 2 network.

- **Key Information (Published on Topics)**:
    - `/joint_states` (`sensor_msgs/JointState`): Current angle and velocity of all robot joints.
    - `/tf` and `/tf_static` (`tf2_msgs/TFMessage`): The transformation tree of the robot's coordinate frames.
    - `/imu/data` (`sensor_msgs/Imu`): Data from the Inertial Measurement Unit.
    - `/diagnostics` (`diagnostic_msgs/DiagnosticArray`): Health and status of hardware and software nodes.

---

## 4. DigitalTwin (Conceptual)

Represents the simulated environment. Like the `HumanoidSystem`, it's not a single data structure but is managed by the simulation software (e.g., Isaac Sim, Gazebo). Its state is exposed to the ROS 2 network through various topics publishing sensor data and object states.
